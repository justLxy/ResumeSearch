## 当前架构概览

当前系统是一个基于 FastAPI 和 Elasticsearch 的简历检索原型，核心链路如下：

1. 解析简历或生成 mock 简历，抽取候选人、教育经历、实习经历、项目经历、技能、投递信息等结构化字段。
2. 在导入阶段生成一路主语义向量：
   - `semantic_profile_vector`：项目、实习、技能、教育方向、目标岗位和专业背景组成的抽取式 profile。
   - 姓名、学校、城市、公司、联系方式、候选人编号、岗位编号等实体字段不进入向量，避免实体查询被语义泛化污染。
3. Elasticsearch mapping 同时保留：
   - `text` 字段用于 IK 分词和 BM25。
   - `keyword` 字段用于精确过滤和实体匹配。
   - `nested` 字段用于教育、实习、项目等一对多结构。
   - `dense_vector` 字段用于 kNN 向量检索。
4. 查询阶段并行执行 BM25 和 `semantic_profile_vector` kNN。
5. 使用应用层手写加权 RRF 融合排序：
   - BM25 负责候选池和词面证据。
   - 向量检索给 BM25 已命中的候选增加 dense rank 贡献。
   - BM25 没命中的 dense-only 候选只有在高置信阈值内才会少量补充。
   - 避免纯语义向量把弱相关候选人自动填满最终结果。

当前设计的核心原则是：实体型查询要尊重精确性，能力型查询要利用语义泛化，混合检索不能让向量泛化破坏实体匹配。

## 一、Elasticsearch 版本与 RRF 能力判断

### 1. 误以为 Basic 版本可以直接使用内置 RRF

**问题背景**

最初希望使用 Elasticsearch 的 `retriever.rrf` 做 BM25 和 kNN 的融合排序，减少自己维护融合逻辑的成本。

**错误表现**

在本地 Basic license 环境下尝试使用内置 RRF retriever 时，ES 返回权限或 license 相关错误，无法正常执行。

**根本原因**

Elasticsearch 的 RRF retriever 并不是 Basic 版本可用能力。本项目本地 ES license 是 Basic，因此不能依赖内置 RRF。

**排查过程**

通过 `_license` 查看当前 license 类型，再用最小查询验证 `retriever.rrf`。确认不是查询 DSL 写错，而是功能授权限制。

**解决方案**

改为手写 RRF：

- BM25 查询返回一组有词面证据的候选人。
- 每个 `dense_vector` 字段单独跑一组 kNN。
- 按 `1 / (rank_constant + rank)` 计算每路排名贡献。
- 用加权 RRF 合并多路结果。

**经验教训**

不要只看 ES 文档里的功能示例，还要确认当前 license、版本和部署模式是否支持。检索系统的能力边界经常受版本、license、插件、mapping 类型共同影响。

**面试表达**

可以这样说：

> 我最初计划直接使用 Elasticsearch 内置 RRF，但在 Basic license 下验证发现不可用。于是我没有继续强依赖高级特性，而是手写了 RRF 融合层。这样一方面规避了 license 限制，另一方面也让我可以控制各路召回的权重和候选过滤策略，后续解决了向量召回污染排序的问题。

## 二、混合检索架构与排序问题

### 2. 等权 RRF 导致向量检索压过 BM25

**问题背景**

系统最初采用 BM25 加 5 路向量检索，再用 RRF 融合。直觉上认为多路召回越多，整体召回越强。

**错误表现**

搜索 `北京大学` 时，返回了只是在期望工作城市或可面试城市里包含“北京”的候选人。比如候选人的学校不是北京大学，甚至教育经历也没有北京大学，但仍然被排到前面。

**根本原因**

当时 RRF 对 BM25 和每路向量等权处理：

- BM25 只有 1 路。
- 向量有 5 路。
- 如果某个候选人在多路向量里都排得较高，即使 BM25 没有命中，也会通过 RRF 被推到最终结果里。

另外，embedding 相似度对中文实体并不可靠。`北京大学` 的向量可能和包含 `北京`、`大学`、城市、教育背景等文本产生较高相似度，但这不等价于实体命中。

**排查过程**

分别打印同一个查询下 BM25 和各路 kNN 的结果：

- `北京大学` 的 BM25 total 为 0。
- 但多路向量仍然返回 0.82 左右的高分候选。
- 说明问题不是 BM25 查错，而是向量召回在没有词面证据的情况下进入了最终融合。

**解决方案**

重新定义混合检索边界：

- BM25 和结构化字段查询决定最终候选池。
- kNN 仍然并行执行，但只允许重排 BM25 已命中的候选。
- RRF 改为加权：
  - BM25 权重更高。
  - 技能、项目、实习、教育等向量根据可靠性赋不同权重。
- `_hybrid_total` 以 BM25 命中数为准，避免把纯向量候选计入有效结果。

**经验教训**

混合检索不是简单把所有召回结果合并。向量召回更适合做语义扩展和重排，实体型查询必须保留词面和结构化字段的约束。否则向量的泛化能力会变成误召回来源。

**面试表达**

可以这样说：

> 我们一开始把 BM25 和多路向量召回等权做 RRF，结果发现实体查询会被向量泛化污染。排查时我把 BM25 和各路 kNN 的结果拆开看，发现 BM25 没命中但向量仍然给出高相似度。后来我把 BM25 改成候选门控，向量只在有词面证据的候选中参与重排，同时给 BM25 更高权重。这既保留了语义排序，又避免实体查询被错误泛化。

### 3. 针对“大学”做特判是错误决策

**问题背景**

为了快速修复 `北京大学` 搜索结果异常，曾经加入过一个判断：如果查询包含 `大学`、`学院`、`学校`、`研究院`，就只查学校字段，跳过向量检索。

**错误表现**

这个方案虽然能修复 `北京大学`，但用户指出它过于刻意。类似实体查询非常多，比如公司、姓名、岗位、项目名称、证书、比赛、实验室名称，不可能为每种词都写特征工程分支。

**根本原因**

这是一个 query pattern 特判，而不是检索架构层面的修复。它把问题从“融合策略不合理”转移成“不断补规则”，维护成本会越来越高。

**排查过程**

重新审视问题后发现，真正的问题不是 `大学` 这个词，而是纯向量候选没有词面证据也能进入最终结果。

**解决方案**

删除学校查询特判，改为通用方案：

- 建立更强的 BM25 结构化查询，包括 `candidate.name.keyword`、`candidate.school.keyword`、`application.position_name.keyword`、`skills`、nested `education.school`、nested `internships.company`、nested `projects.name` 等。
- BM25 命中的候选才进入最终 RRF 候选池。
- 向量检索只在这些候选中贡献排序分。

**经验教训**

当一个问题有很多同类变体时，不要用特判补丁。应该抽象出问题背后的检索原则：哪些字段承担精确匹配，哪些字段承担语义泛化，哪些模块负责候选召回，哪些模块负责排序。

**面试表达**

可以这样说：

> 我曾经为了快速解决学校查询误召回，加过“包含大学就走学校字段”的特判。但后来意识到这是错误方向，因为公司名、项目名、人名也会遇到同类问题。最后我撤掉特判，改成通用的 BM25 候选门控和向量重排，这才是可扩展的混合检索设计。

### 4. “无论搜什么都约 50 条”的问题

**问题背景**

系统为了增加召回，把 `rank_window_size` 和 kNN 候选数设置得比较大，同时前端展示了结果数。

**错误表现**

用户搜索不同关键词时，页面显示“约 50 条结果”，看起来像是搜索条件没有生效。

**根本原因**

有两个问题叠加：

- 后端把多路召回合并后的候选数量当成匹配总数，向量召回很容易凑满窗口。
- 前端展示的是返回数量或候选窗口数量，而不是有效词面匹配数量。

**排查过程**

对比 `hits.total`、RRF 合并后的唯一文档数、前端展示数量，发现“约 50 条”来自检索窗口和向量候选数，而不是业务意义上的匹配总数。

**解决方案**

- 对有查询词的混合检索，`candidate_total` 使用 BM25 命中数。
- 纯向量候选不计入有效匹配总数。
- 前端区分 `returned_count`、`matched_total`、`candidate_total`。

**经验教训**

搜索系统里的“总数”要定义清楚：

- 命中总数：满足查询条件的文档数。
- 返回数量：当前页面实际返回的文档数。
- 候选窗口：参与排序的内部候选数。

这些概念不能混用，否则 UI 会给用户错误反馈。

**面试表达**

可以这样说：

> 这个问题看起来是前端显示 bug，实际是检索语义不清。向量召回会固定拿一批候选，如果把候选窗口当成命中数，用户就会觉得搜索条件无效。我后来把 BM25 命中数作为有效匹配总数，返回数量单独展示，内部候选窗口不暴露给用户。

### 5. 空查询和默认搜索只显示 20 条，而不是完整结果窗口

**问题背景**

搜索接口的 `limit` 默认是 20。空查询本来希望作为“浏览全部候选人”入口，后来关键词搜索也希望默认返回当前结果窗口内的完整候选集。

**错误表现**

页面什么都不输入或直接搜索时，只显示 20 条。用户预期是看到所有候选人或至少看到当前系统支持的全部浏览结果。

**根本原因**

接口默认 `limit=20`，前端没有显式传 `limit`，所以所有搜索都会吃这个默认值。

**排查过程**

查看 `/api/search` 的默认参数和空查询分支，发现空查询也受前端 limit 影响。

**解决方案**

当前实现把默认 `limit` 改为 0，表示使用 `MAX_BROWSE_RESULT_SIZE` 作为结果窗口。空查询、只有筛选条件和关键词混合检索在未显式传 `limit` 时都会尽量返回当前窗口内的全部候选人；如果调用方显式传 `limit=20`，仍然只返回 20 条。

**经验教训**

搜索和浏览是两个不同用户意图。空查询不应该简单复用关键词搜索的默认分页参数，尤其在候选人数量较小时。

**面试表达**

可以这样说：

> 我把未显式分页的搜索看成“返回当前结果窗口”场景，而不是固定 20 条的分页场景。当前用 `MAX_BROWSE_RESULT_SIZE` 做技术边界，避免 API 构造真正无限的 ES/kNN/RRF 窗口。

### 6. 多级层级（Exact/Phrase/Term）同时命中导致的算分膨胀

**问题背景**

为了保证召回和排序的层次感，同一个字段（如学校）会构造精确匹配、短语匹配和普通词项匹配三个层级的查询。最初这些查询都被放在一个 `bool.should` 中。

**错误表现**

当候选人的学校和查询词完全一致时，Elasticsearch 会同时命中这三个层级的查询，`bool.should` 会把这三者的分数累加（Sum），导致算分异常膨胀，打破了与其他单次命中特征的权重平衡。

**解决方案**

将打分查询由 `bool.should` 改为 `dis_max`，并设置 `tie_breaker: 0.0`。这样，当一个字段同时命中多个层级时，只取最高层级（最大权重）的分数，而不再累加。

**面试表达**

可以这样说：
> 在处理多层级检索（精确、短语、普通匹配）时，一开始直接用了 bool.should，结果发现完全匹配的文档分数膨胀得厉害，因为它同时吃到了三个查询的加分。我后来改用了 dis_max 策略，同一组语义特征下只取最高得分，有效地控制了算分的边界，保持了权重的稳定性。

### 7. 嵌套查询（Nested Query）导致底层权重被掩盖与前端标签重复

**问题背景**

原本在后端为 `nested` 查询打标时，标签只打在了最外层的 `nested` 包装器上（如 `[精确匹配] 教育经历`），而不带具体的字段和权重。同时由于 `dis_max` 依然会返回所有命中的子查询，前端出现了诸如 `[短语匹配] 学校` 和 `[精确匹配] 学校` 同时存在的重复标签。

**解决方案**

1. **后端透出权重**：移除 `nested` 外层的打标，将带有权重的标签（如 `lexical_term:教育经历-实验室:W2`）直接下推到最底层的 `match` 或 `multi_match` 查询上。
2. **前端去重折叠**：在前端渲染命中详情时，解析出字段名（如“学校”），并根据匹配层级优先级（精确 > 短语 > 普通）对相同字段名进行去重，只展示最高优的带权重标签。

### 8. 单 Token 查询时“覆盖数”显示为 0 的疑惑

**问题背景**

当搜索没有空格的单一名词（如“智能信息处理实验室”）时，用户发现命中详情里的“覆盖数”为 0，以为是系统漏算了覆盖度。

**根本原因**

覆盖度（Coverage）机制的初衷是奖励“跨字段命中多个不同搜索词”的简历。如果搜索词本身只由一个连续的 Token 组成（拆分后 `len(tokens) < 2`），在任何字段命中都只是常规匹配，不存在“多词分散命中”的覆盖度奖励，因此系统故意跳过了覆盖度的计算（显示为 0）。

**解决方案**

向用户解释该算分机制的业务逻辑，不视为 Bug，但在遇到类似反馈时需要能清晰地指出这是为长句、多条件（如 `北京 Java 架构师`）设计的专属加分项。

## 三、向量化字段设计与 embedding 问题

### 6. 把实体字段放进向量导致语义污染

**问题背景**

为了让向量包含更多信息，早期把候选人姓名、学校、当前城市、期望城市、公司等字段都拼进了向量文本。

**错误表现**

搜索 `北京大学` 时，结果里出现了和学校无关、只是城市字段包含北京的候选人。搜索公司、城市、学校等实体时也存在潜在误召回风险。

**根本原因**

向量模型擅长语义相似，不擅长保证实体精确匹配。实体字段进入向量后，会把 `北京大学`、`北京`、`大学`、学校、城市等概念混在同一个语义空间里，导致向量相似不等于业务相关。

**排查过程**

检查 `_build_search_text` 和 `_embedding_inputs`，发现向量输入包含大量结构化实体信息。进一步抽样查看索引中的 `search_text`，确认学校、城市、公司名曾经进入语义向量文本。

**解决方案**

重新定义向量字段职责：

- `search_text_vector` 只保留岗位、专业、技能、教育语义、实习语义、项目语义。
- `education_vector` 排除学校、学院、城市，只保留学历、学位、专业、研究方向、实验室、论文等级。
- `internships_vector` 排除公司名，只保留部门、岗位、工作性质、工作描述，并对描述中的公司、城市等实体做清洗。
- `projects_vector` 保留项目名称、描述、职责，因为项目名称通常也是能力场景的一部分。
- `skills_vector` 只保留技能标签。

同时增加 `_semantic_exclusions` 和 `_strip_semantic_exclusions`，从语义向量文本中剔除姓名、学校、城市、公司等结构化实体。

**经验教训**

向量化不是“把所有文本拼起来”。应该先区分字段类型：

- 实体字段：姓名、学校、公司、城市，适合 keyword、phrase、nested 精确查。
- 语义字段：职责、项目描述、研究方向、技能，适合 embedding。
- 过滤字段：学历、城市、年限，适合 keyword/range filter。

**面试表达**

可以这样说：

> 这个项目里我最大的经验是，向量字段不是越全越好。最初我把学校、城市、公司都放进向量，结果实体查询被语义泛化污染。后来我把实体字段从 embedding 输入中剥离，让 BM25 和 keyword 负责精确匹配，让向量只表达能力和经历语义。这是混合检索里非常关键的字段职责划分。

### 7. embedding 维度和归一化必须和 mapping 保持一致

**问题背景**

项目使用 `IEITYuan/Yuan-embedding-2.0-zh` 生成中文 embedding，并写入 ES `dense_vector`。

**错误表现**

如果 embedding 输出维度和 ES mapping 的 `dims` 不一致，写入会失败。归一化策略不一致也会导致查询向量和文档向量相似度不可比。

**根本原因**

ES `dense_vector` 的维度在索引创建时固定，不能动态变化。项目中模型输出维度是 1792，因此 mapping 中 `dims` 必须是 1792。文档向量和查询向量都必须使用同一个模型、同一个 pooling、同一个归一化策略。

**排查过程**

统一在 `embedding_service.py` 中定义 `VECTOR_DIMS=1792`，导入时检查每个向量长度。如果长度不等于 `VECTOR_DIMS`，直接抛出异常。

**解决方案**

- mapping 里的 `dense_vector.dims` 使用 `VECTOR_DIMS`。
- 文档向量和查询向量都调用同一个 `encode` 方法。
- `encode` 使用 `normalize_embeddings=True`。
- ES mapping 使用 `similarity: cosine`。

**经验教训**

embedding 模型是索引 schema 的一部分。更换模型、维度、归一化策略后必须重建索引，不能只改查询侧。

**面试表达**

可以这样说：

> 我把 embedding 维度抽成统一常量，并在导入时校验向量长度。因为 ES 的 dense_vector 维度是索引级 schema，一旦模型换了或输出维度变了，必须重建索引。查询向量和文档向量也必须用同一模型和归一化策略，否则相似度没有可比性。

### 8. ModelScope 模型目录结构与 Dense 权重问题

**问题背景**

本地通过 ModelScope 下载中文 embedding 模型，但 `sentence_transformers` 对模型目录结构有固定预期。

**错误表现**

ModelScope 下载后文件结构和 `sentence_transformers` 期望的 `1_Pooling/`、`2_Dense/` 子目录不完全一致，可能导致模型加载失败或输出维度不正确。

**根本原因**

ModelScope 把部分文件平铺到了模型根目录，但 `sentence_transformers` 会根据模块目录加载 pooling 和 dense 层配置。Dense 层权重还需要从 HuggingFace 或镜像补齐。

**排查过程**

查看模型加载报错和模型目录结构，发现缺少 `1_Pooling/config.json`、`2_Dense/config.json` 和 Dense 层权重。

**解决方案**

在 `embedding_service.py` 中：

- 下载 ModelScope 模型。
- 重建 `1_Pooling/` 和 `2_Dense/` 目录。
- 写入 pooling 和 dense 配置。
- 从 HuggingFace 或 `hf-mirror.com` 下载 Dense 权重。
- 使用 `local_files_only=True` 加载本地模型。

**经验教训**

模型接入不能只验证“能下载”，还要验证“加载结构、输出维度、归一化策略、批量编码”都正确。模型加载逻辑最好集中在一个模块里，避免导入和查询侧各写一套。

**面试表达**

可以这样说：

> 这个模型接入时遇到过 ModelScope 和 sentence-transformers 目录结构不一致的问题。我最后在 embedding service 里统一修复目录结构，补齐 pooling 和 dense 配置，并用维度校验保证输出是 1792 维。这样导入和查询都复用同一套编码逻辑。

## 四、Mapping、IK 分词与结构化字段设计

### 9. 只依赖全文字段会丢失结构化语义

**问题背景**

简历里有大量结构化信息，如教育经历、实习经历、项目经历、技能、学历、城市、年限。早期检索更偏向把文本拼成大字段。

**错误表现**

字段语义不清晰，学校、公司、项目、技能、学历等不同类型的查询难以分别调权。过滤和排序也容易混在全文检索里。

**根本原因**

简历不是普通文章，而是半结构化文档。不同字段对检索的意义不同：

- 学校、公司、姓名需要精确匹配。
- 项目职责和实习描述需要语义匹配。
- 学历、城市、技能、年限需要过滤。
- 教育和项目是一对多结构，需要 nested 查询。

**排查过程**

从用户查询行为倒推字段职责，逐步把候选人、投递、教育、实习、项目、技能拆成不同 mapping。

**解决方案**

mapping 中同时使用：

- `keyword`：用于 ID、学历、城市、技能、精确实体。
- `text + IK analyzer`：用于中文 BM25。
- `fields.keyword`：用于同一字段的精确匹配和短语匹配。
- `nested`：用于教育、实习、项目数组，避免跨对象误匹配。
- `dense_vector`：用于语义检索。

**经验教训**

搜索系统的 mapping 设计决定了后续能不能调优。越早把字段职责建模清楚，后续越少靠补丁修问题。

**面试表达**

可以这样说：

> 我没有把简历简单当成一段全文，而是拆成 candidate、application、education、internships、projects、skills 等结构。精确实体走 keyword，中文文本走 IK BM25，一对多经历走 nested，语义字段走 dense_vector。这样后续可以分别控制召回、过滤和排序。

### 10. IK 分词要区分索引分词和搜索分词

**问题背景**

中文简历文本需要分词后才能做 BM25。项目使用 IK 分词器。

**错误表现**

如果索引和查询都使用同一种分词策略，可能出现召回不足或噪声过多。比如 `ik_max_word` 召回强但词粒度太细，`ik_smart` 更保守但可能漏召回。

**根本原因**

索引阶段和查询阶段目标不同：

- 索引阶段希望保留更多词元，提高潜在召回。
- 查询阶段希望减少过度拆分，降低噪声。

**排查过程**

检查 analyzer 设置后，将简历正文类字段配置为索引用 `resume_text`，搜索用 `resume_search`。

**解决方案**

- `resume_text` 使用 `ik_max_word`。
- `resume_search` 使用 `ik_smart`。
- 学校、姓名、岗位等字段额外保留 `.keyword` 子字段，避免完全依赖分词。

**经验教训**

中文检索中，分词召回和实体精确匹配要结合使用。对学校名、公司名、人名这类实体，不能只依赖分词。

**面试表达**

可以这样说：

> 我用 IK 做中文 BM25，但没有只依赖全文分词。索引时用 ik_max_word 提高召回，搜索时用 ik_smart 降低噪声，同时给实体字段保留 keyword 子字段，用于精确匹配和高权重 phrase 查询。

## 五、数据解析、导入、索引构建与同步问题

### 11. HTML `.doc` 简历解析不能当作普通文本处理

**问题背景**

真实简历文件后缀是 `.doc`，但内容实际是 HTML 表格。需要从里面抽取个人信息、教育经历、实习经历、项目经验、语言能力、IT 技能、奖励活动等结构化数据。

**错误表现**

如果只把文件当成普通文本读取，会出现几个问题：

- 字段边界不稳定，无法区分“学校”“公司”“项目名称”等字段。
- 教育、项目、奖励等重复段落会被合并成数组或丢失上下文。
- `至今`、`Present` 这类日期不能直接写入 date 字段。
- 有些文件编码可能是 UTF-8、GBK、GB18030，直接按单一编码读取容易失败。
- 原始 `.doc` 可能不是 HTML 格式，解析器需要能识别并失败隔离。

**根本原因**

这类简历不是标准 Word 二进制文档，也不是自然语言长文本，而是 HTML table 形式的半结构化表单。它既有固定字段，又有重复段落，还存在中文字段名、空值、不同日期格式、不同 section 标题写法。

**排查过程**

先读取原始文件内容，确认里面有 `<html>` 和 `<tr>/<td>`。随后按表格行检查，发现每一行通常是“字段名 + 字段值”，section 标题如 `个人信息`、`教育经历`、`项目经验` 出现在独立行。重复 section 则通过 `开始日期`、`IT技能`、`是否有参赛获奖经历` 等起始字段区分。

**解决方案**

在 `resume_parser.py` 中实现专门解析器：

- `_decode_html_doc` 检测 HTML `.doc`，按声明 charset、`utf-8-sig`、`utf-8`、`gb18030` 依次尝试解码。
- `_extract_section_pairs` 按 section 标题切分表格内容。
- `_map_repeating_section` 根据起始字段拆分教育、实习、项目、奖励、IT 技能等重复记录。
- `_normalize_date` 将可解析日期转为 ISO date，将 `至今`、`现在`、`present` 归一成 `None`，并保留 `*_raw`。
- `parse_resume_batch` 对单文件解析失败做隔离，不让一个坏文件中断整个批次。
- 保留 `raw_sections` 和 `raw_fields` 方便调试，但导入 ES 前清理掉调试字段，避免索引膨胀。

**经验教训**

半结构化简历解析不能靠简单正则或全文切分。应该先识别文档真实格式，再保留结构边界。重复 section 的切分键、日期归一化、编码探测、失败隔离都是导入质量的基础。

**面试表达**

可以这样说：

> 真实简历虽然是 `.doc` 后缀，但实际是 HTML 表格。我没有直接做全文抽取，而是先按 section 标题切分，再把每个 section 的表格行转成字段键值对。教育、项目、奖励这类重复块通过起始字段拆分，日期里 `至今` 会归一成空值并保留 raw 字段。这样后续 ES mapping、nested 查询和向量化都有稳定结构。

### 12. 用内容 hash 作为 resume_id 会影响更新语义

**问题背景**

早期解析真实简历时，`resume_id` 使用文件内容 hash。这样可以保证同一文件内容对应同一个 ID。

**错误表现**

如果候选人同一个人更新了简历内容，hash 会变化，系统会把它当成新文档插入。旧文档如果没有删除逻辑，就会产生重复候选人。

**根本原因**

内容 hash 适合文件去重，不适合作为业务主键。简历检索的主键应该是候选人编号、投递编号或稳定业务 ID。

**排查过程**

检查导入后的 `_id` 和业务字段，发现真实简历里有 `application.candidate_no`，mock 数据也可以生成稳定候选编号。

**解决方案**

mock 导入使用候选人编号作为 `resume_id`。真实导入链路也应优先使用业务稳定 ID，只有缺失时才退回到文件 hash。

**经验教训**

索引文档 `_id` 是更新和删除同步的基础。业务实体主键和文件内容指纹不能混用。

**面试表达**

可以这样说：

> 我最开始用文件 hash 当文档 ID，这对去重方便，但不利于更新。候选人更新简历后 hash 变了，会产生新文档。后来我把候选人编号作为稳定 ID，把文件 hash 只作为元数据，这样才能支持覆盖更新和删除同步。

### 13. 重建索引需要 versioned index 加 alias

**问题背景**

mapping、analyzer、dense_vector 维度或向量字段设计变更后，都需要重建索引。

**错误表现**

如果直接在原索引上删除重建，导入失败会导致线上 alias 不可用，或者部分数据被写入但查询结果不完整。

**根本原因**

ES mapping 很多字段不能原地修改，重建索引是常态。直接操作线上索引缺少回滚空间。

**排查过程**

结合多次 mapping 和向量字段变更，发现需要一种安全的 rebuild 流程。

**解决方案**

实现 versioned index：

1. 创建 `resumes_v1_时间戳` 新索引。
2. 全量 bulk 写入。
3. refresh 并校验 count。
4. 成功后原子切换 `resumes_current` alias。
5. 失败时 alias 仍指向旧索引。

**经验教训**

搜索索引应该把“物理索引”和“逻辑访问名”分离。业务查询只访问 alias，重建过程使用新索引，成功后再切换。

**面试表达**

可以这样说：

> 因为 ES mapping 和向量维度变更都需要重建索引，所以我做了 versioned index 加 alias 的机制。导入先写新索引并校验数量，成功后再原子切 alias。这样即使导入失败，线上查询仍然指向旧索引。

### 14. 本地 ES 磁盘水位导致新索引分片无法分配

**问题背景**

在多次重建索引时，本地机器磁盘使用率超过 ES 默认 high watermark。

**错误表现**

新索引创建后处于 red 状态，bulk 写入时报：

```text
unavailable_shards_exception: primary shard is not active
disk_threshold: node is above the high watermark
```

**根本原因**

ES 默认不允许在磁盘超过 high watermark 的节点上分配新分片。即使索引本身很小，只要节点磁盘水位超过阈值，也会拒绝新分片分配。

**排查过程**

执行：

```bash
GET /_cluster/health
GET /_cat/indices/resumes_v1*
GET /_cluster/allocation/explain
```

allocation explain 明确显示 `disk_threshold` 拒绝分片分配，剩余磁盘低于 ES 要求。

**解决方案**

本地开发环境中临时关闭磁盘分配阈值：

```bash
PUT /_cluster/settings
{
  "transient": {
    "cluster.routing.allocation.disk.threshold_enabled": false
  }
}
```

导入成功并切换 alias 后恢复：

```bash
PUT /_cluster/settings
{
  "transient": {
    "cluster.routing.allocation.disk.threshold_enabled": null
  }
}
```

同时删除失败的 red 临时索引，确保集群恢复 green。

**经验教训**

本地调试可以临时关闭磁盘阈值，但生产环境不能这么做。生产环境应该清理磁盘、扩容数据节点、调整 ILM 或迁移旧索引。

**面试表达**

可以这样说：

> 重建索引时遇到过 bulk 写入 503，表面看是 shard not active。通过 allocation explain 定位到本机磁盘超过 high watermark，ES 不分配新主分片。本地我临时关闭阈值完成导入并马上恢复，生产环境则应该通过扩容或清理磁盘解决，而不是关闭保护。

### 15. 导入请求 30 秒超时太短

**问题背景**

创建包含多个 HNSW `dense_vector` 字段的新索引时，ES 本地响应偶尔超过 30 秒。

**错误表现**

导入脚本在 `PUT /index` 或 bulk 阶段出现 `ReadTimeout`。有时索引已经部分创建，但脚本认为失败。

**根本原因**

HNSW 向量字段初始化和本地 ES 状态受磁盘、水位、JVM、分片分配影响。30 秒对索引重建流程偏短。

**排查过程**

导入失败后检查 `_cat/indices`，发现有 red 临时索引残留，说明请求超时和 ES 异步状态变化叠加。

**解决方案**

- 将导入脚本通用请求超时提高到 90 秒。
- 导入失败后检查并清理半成品索引。
- 保持 alias 先不切换，避免失败索引影响线上查询。

**经验教训**

批量导入和索引重建不是普通在线查询，超时应该单独配置。失败后要检查外部系统状态，不能只依赖脚本异常。

**面试表达**

可以这样说：

> 我把在线查询和离线导入的超时区分开。查询要短超时保护用户体验，但导入创建 HNSW 索引可能更慢，所以导入侧超时提高到 90 秒，并且失败后会检查是否留下半成品索引。

### 16. 增量删除要谨慎处理空数据集

**问题背景**

导入脚本支持 `delete_missing`，用于删除本次导入数据里不存在的旧文档。

**错误表现**

如果解析结果为空但仍执行 delete missing，可能误删整个索引。

**根本原因**

增量同步里“本次没有解析到文档”可能代表数据源异常，而不是业务上所有文档都删除。

**排查过程**

审查导入逻辑时发现，需要在 recreate 和 delete missing 两个场景分别处理空数据。

**解决方案**

- recreate 模式下如果没有解析到文档，直接中止重建。
- `_delete_missing_docs` 中如果 live_ids 为空则直接 return。

**经验教训**

删除同步永远要比写入更保守。空数据集不能默认代表全量删除。

**面试表达**

可以这样说：

> 增量同步里我特别处理了空数据集。因为解析失败和业务删除在表现上都可能是本次数据为空，如果直接 delete_by_query 会误删全量索引。所以我让空 live_ids 直接跳过删除，全量重建时没有 parsed docs 也会直接失败。

## 六、筛选、计数和前端展示问题

### 17. 学历筛选“本科、学士、硕士都是 20 人”

**问题背景**

前端提供学历筛选按钮，包括本科、学士、硕士等。

**错误表现**

用户选择不同学历后，页面仍然显示 20 人，看起来筛选条件没有生效。

**根本原因**

存在两层问题：

- UI 默认只显示返回条数，历史上受 `limit=20` 限制，容易让不同筛选看起来都是 20。
- 数据中 `学士` 是学位，`本科` 是学历层次；如果直接拿 `学士` 去匹配 `candidate.highest_degree`，语义不一致。

**排查过程**

查看 mapping 和数据后确认 `candidate.highest_degree` 存的是 `本科` 或 `硕士`。前端传 `学士` 时，后端需要归一化。

**解决方案**

- 后端 `_normalize_highest_degree` 把 `学士` 归一化为 `本科`，把 `硕士研究生` 归一化为 `硕士`。
- 展示层区分返回条数和匹配总数。
- 默认搜索和筛选浏览场景返回当前 `MAX_BROWSE_RESULT_SIZE` 窗口内的结果，减少“总是 20”的错觉。

**经验教训**

筛选项必须和索引字段的业务语义一致。学历、学位、学制是不同概念，不能混成同一个字段。

**面试表达**

可以这样说：

> 这个问题不是 ES 查不准，而是字段语义和 UI 展示混在一起了。学士是学位，本科是学历，我在后端做了归一化，同时让前端不要把当前页返回数量当成匹配总数展示。

### 18. mock 数据区分度不足影响检索效果判断

**问题背景**

为了演示和测试，需要生成 50 份 mock 简历。

**错误表现**

早期 mock 候选人的背景过于相似。比如项目职责、技能、实习经历都使用模板化文本，导致搜索不同关键词时结果差异不明显。

**根本原因**

检索系统的效果很依赖数据分布。如果 mock 数据同质化，BM25 和向量都难以拉开差距，也不利于观察排序质量。

**排查过程**

从页面结果观察到多个候选人片段高度重复，技能标签和项目经历差异很小。

**解决方案**

在 `generate_mock_resumes.py` 中按岗位建立 `POSITION_BACKGROUNDS`：

- 每类岗位有不同部门、研究方向、实验室、项目、实习动作、结果指标。
- 生成项目和实习时根据岗位 profile 取样。
- 保证机器学习、后端、安全、数据分析、前端、测试、运维、算法、Java、安全运营等岗位具备明显差异。

**经验教训**

mock 数据不是随便造字段。用于检索评估的 mock 数据必须有可区分的主题、实体和能力标签，否则无法判断算法是否有效。

**面试表达**

可以这样说：

> 我后来专门优化了 mock 数据，让不同岗位有不同技能、项目、实习动作和结果指标。否则检索系统看起来总是返回相似候选人，很难判断是算法问题还是数据同质化问题。

## 七、性能、资源与稳定性问题

### 19. 多路向量检索带来查询成本上升

**问题背景**

历史版本为了提升语义召回，曾使用 5 个向量字段，并行执行 5 路 kNN 加 1 路 BM25。当前实现已经收敛为 1 路 `semantic_profile_vector`。

**错误表现**

每次关键词搜索都会产生 6 个 ES 请求。随着数据量增加，查询延迟、CPU、内存和 HNSW 图检索成本都会上升。

**根本原因**

多路向量检索提高了表达能力，但也线性增加了查询开销。`num_candidates` 越大，召回越充分，成本也越高。

**排查过程**

审查 `_run_hybrid_search` 和 `_knn_body`，确认每个字段都会单独请求 ES，`num_candidates` 由 `size` 和 `KNN_NUM_CANDIDATES` 共同决定。

**解决方案**

当前开发环境采用：

- 并行执行 BM25 和 1 路 kNN，降低串行等待时间。
- `_source` 排除大字段和向量字段，减少网络传输。
- `rank_window_size` 和 `KNN_NUM_CANDIDATES` 设置上限。
- facets 和向量字段列表做缓存。

后续生产优化方向：

- 做离线评估，决定是否继续保留单一 profile，还是拆分出少量高收益向量字段。
- 对短 query、实体 query 降低向量参与度。
- 引入 reranker 时减少初召窗口。
- 使用 pagination/search_after，避免大窗口深翻页。

**经验教训**

多路召回不是免费午餐。每增加一路召回，都要问它是否显著提升效果，是否值得增加延迟和资源成本。

**面试表达**

可以这样说：

> 我早期尝试过多路向量表达技能、项目、实习、教育等不同语义面，但每路 kNN 都有成本。后来收敛成一路抽取式 `semantic_profile_vector`，查询侧只并行 BM25 和 1 路 kNN，同时排除大字段和向量字段，并控制 rank window 和 num_candidates。后续是否拆分向量字段要由评估集决定。

### 20. `_source` 返回大字段和向量会浪费网络与内存

**问题背景**

ES 文档中包含原始文本、分段文本、搜索拼接文本和 1792 维语义向量。

**错误表现**

如果查询结果直接返回完整 `_source`，单次请求会传输大量不需要展示的数据，影响响应速度和前端内存。

**根本原因**

向量字段很大，原始简历文本也很长。检索列表页只需要候选人摘要、教育摘要、技能、片段和少量元信息。

**排查过程**

检查查询响应体，发现向量字段和 raw text 对列表页没有价值。

**解决方案**

统一定义 `SOURCE_EXCLUDES`，排除：

- `raw_text`
- `raw_sections`
- `search_text`
- `skills_text`
- 所有 vector 字段

**经验教训**

检索系统要区分列表页、详情页和调试接口。列表页返回越小越好，详情页再按需取完整文档。

**面试表达**

可以这样说：

> 我在列表查询里排除了 raw text、拼接文本和所有向量字段。因为 1792 维向量和原始简历对列表展示没有意义，只会增加网络和内存开销。详情页再单独按 ID 获取完整信息。

## 八、代码结构与可维护性

### 21. 查询逻辑需要模块化，不能散落在接口里

**问题背景**

搜索接口包含过滤、BM25、kNN、RRF、格式化、facets 等逻辑。

**错误表现**

如果所有逻辑都写在 `/api/search` 里，每次调参都会影响主流程，可读性和测试性都差。

**根本原因**

检索系统变化频繁，query builder、融合算法、格式化逻辑需要独立演进。

**排查过程**

随着问题增多，逐步把函数拆出来：

- `_build_filters`
- `_bm25_body`
- `_lexical_query`
- `_knn_body`
- `_run_hybrid_search`
- `_rrf_merge`
- `_format_hit`

**解决方案**

将接口层保持为编排逻辑，具体检索逻辑放入独立函数。测试中直接调用关键函数验证行为。

**经验教训**

检索项目不要把 DSL 拼装、融合排序、展示格式化混在一起。模块化后，调参和补测试都更容易。

**面试表达**

可以这样说：

> 我把搜索接口拆成 query builder、retriever、fusion、formatter 几层。这样后续改 RRF 权重、字段 boost、向量字段，不需要改接口主流程，也能单独写测试覆盖融合行为。

### 22. 必须为检索策略写行为测试

**问题背景**

检索效果问题往往不是语法错误，而是策略行为不符合预期。

**错误表现**

如果没有测试，后续很容易再次把纯向量候选放进最终结果，或者把实体字段重新拼进向量文本。

**根本原因**

搜索策略通常是多个规则叠加，手工验证容易遗漏回归。

**排查过程**

针对已经踩过的坑补测试：

- 纯向量候选不能单独进入最终结果。
- 有 BM25 词面证据的候选可以被向量重排。
- 向量文本不能包含学校、城市、姓名、公司等实体噪声。
- 实习年限只统计实习经历，不使用项目经历代替工作经验。

**解决方案**

在 `tests/test_search_logic.py` 中增加行为测试，当前测试为 `23 passed`。

**经验教训**

检索系统测试不只测函数输出，还要测策略边界。尤其是“哪些结果不能出现”非常重要。

**面试表达**

可以这样说：

> 我给检索策略补了单元测试，不只是测试正常返回，还测试负面约束。比如纯向量命中的文档不能进最终结果，实体字段不能进向量文本。这样能防止后续调参时把已经修过的问题重新引入。

## 九、当前仍需注意的风险

### 1. BM25 门控可能降低纯语义查询召回

当前方案要求最终结果必须有 BM25 词面证据。这能解决实体查询误召回，但可能降低完全同义表达的召回。例如用户输入“算法落地经验”，如果简历只写“模型部署和离线评估”，BM25 可能不命中。

后续优化方向：

- 建立查询分类，不靠关键词特判，而是根据 BM25 命中强度、query 长度、实体识别结果动态调整。
- 当 BM25 命中数过少时，允许少量高置信向量候选进入候选池，但需要阈值和解释字段。
- 引入重排模型，对 BM25 和向量候选做 cross-encoder rerank。

### 2. 缺少系统化评估集

目前主要靠人工 case 验证，如 `北京大学`、`北京交通大学`、`机器学习 医疗`、`医疗推荐算法经验`。

后续应该建立评估集：

- 实体查询：学校、公司、姓名、项目名、岗位名。
- 能力查询：技能、项目职责、业务场景。
- 组合查询：技能加行业、学校加专业、岗位加年限。
- 负例查询：不存在的学校、公司、技能组合。

指标包括 Recall@K、Precision@K、MRR、NDCG、无结果准确率、响应延迟。

### 3. 旧索引需要清理策略

versioned index 保证了重建安全，但多次重建会留下多个旧索引。本地已经出现多个 `resumes_v1_时间戳` 索引。

后续需要：

- 保留最近 N 个可回滚索引。
- 定期删除更老索引。
- 删除前确认 alias 不指向该索引。

### 4. 生产环境不应关闭磁盘阈值

本地为了完成开发验证，临时关闭过 `cluster.routing.allocation.disk.threshold_enabled`。生产环境必须通过扩容、清理磁盘、调整索引生命周期来解决。

## 十、面试回答框架

### 项目难点怎么讲

可以按这个结构回答：

> 这个项目的难点不是简单把 ES 和 embedding 接起来，而是要处理结构化实体匹配和语义匹配之间的冲突。简历里既有学校、公司、姓名这种必须精确的实体，也有项目职责、技能经验这种适合语义泛化的文本。我一开始把 BM25 和多路向量等权融合，结果实体查询被向量召回污染。后来我把字段职责重新拆分，实体走 keyword、phrase、nested BM25，能力描述走 embedding，并用 BM25 做候选门控、向量做重排，最终解决了相关性问题。

### 问题排查怎么讲

可以按这个结构回答：

> 我排查搜索问题时不会只看最终结果，而是把 BM25、每一路 kNN、RRF 融合结果拆开看。比如 `北京大学` 这个 case，BM25 total 是 0，但多路向量仍然返回高相似候选，所以我定位到问题在融合策略和向量输入，而不是分词或 mapping。之后我检查索引里的 `search_text`，确认城市、学校、公司等实体进了向量文本，于是做了向量字段重构和实体清洗。

### 架构演进怎么讲

可以按这个结构回答：

> 架构经历了三个阶段。第一阶段是 BM25 关键词检索，能解决精确词面匹配。第二阶段加入 dense_vector 和 kNN，提高能力类查询的语义召回。第三阶段加入手写加权 RRF，并重新定义 BM25 和向量的职责：BM25 负责候选池和实体精确性，向量负责有词面证据候选的语义重排。这个演进过程解决了召回和精确性之间的平衡问题。

### 工程化经验怎么讲

可以按这个结构回答：

> 工程上我做了几个保护：第一，embedding 维度和 mapping 共用常量，导入时校验向量长度；第二，重建索引用 versioned index 加 alias，成功后再切换，失败不影响当前查询；第三，批量导入提高超时并处理半成品索引；第四，查询返回排除 raw text 和 vector，减少网络开销；第五，对检索策略补行为测试，防止向量候选绕过 BM25 门控。

## 十一、后续开发检查清单

修改检索逻辑前，先检查：

- 新字段属于实体、语义文本，还是过滤条件。
- 实体字段是否有 `keyword` 或 phrase 查询。
- 一对多字段是否需要 nested。
- 新字段是否真的应该进入 embedding。
- 查询结果总数展示的是命中数、返回数，还是候选窗口。
- 向量候选是否可能绕过词面证据进入最终结果。
- RRF 权重是否有评估依据。
- 导入后是否校验 count、alias、vector exists。
- 重建失败后是否会留下 red 索引。
- 是否需要更新测试覆盖新策略。
